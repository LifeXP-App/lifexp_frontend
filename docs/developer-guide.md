# LifeXP Frontend Developer Guide

This guide is the codebase-level map for developers working on the LifeXP web frontend. It explains the app architecture, routing model, auth flow, backend integration patterns, Convex real-time session system, and the conventions to follow when adding features.

## Quick Start

LifeXP web is a Next.js App Router application with TypeScript, React, Tailwind CSS, Convex, and a Django REST backend.

Required environment variables:

```bash
NEXT_PUBLIC_API_BASE_URL=http://127.0.0.1:8000
NEXT_PUBLIC_CONVEX_URL=https://your-convex-deployment.convex.cloud
```

Common commands:

```bash
npm install
npm run dev
npm run lint
npm run build
npx convex dev
npx convex deploy
```

Notes:

- The repo currently contains both `package-lock.json` and `yarn.lock`. Pick one package manager for a change and avoid lockfile churn unless intentionally cleaning this up.
- `convex/_generated/*` is generated by Convex. Do not hand-edit those files.
- `.env.local` is local-only and should not be committed.

## Product Model

LifeXP is a gamified personal growth platform. Users create goals, run sessions against activities, and earn XP across five aspects:

| Aspect | Meaning | Mastery title |
| --- | --- | --- |
| `physique` | Fitness, movement, physical resilience | Warrior |
| `energy` | Motivation, mood, stamina, focus | Protagonist |
| `logic` | Learning, strategy, analysis | Prodigy |
| `creativity` | Art, experimentation, creation | Alchemist |
| `social` | Communication, collaboration, empathy | Diplomat |

The strongest aspect influences the user's mastery title and visual accent system. Shared aspect metadata lives in:

- `src/lib/constants/aspects.ts`
- `src/lib/types/activityMeta.tsx`
- `app/globals.css`

## Routing Model

The app uses Next route groups to separate regular authenticated screens from fullscreen flows.

### Main Shell

Routes under `app/(main)` render inside:

- `app/(main)/layout.tsx`
- `src/components/Sidebar.tsx`
- `src/context/AuthContext.tsx`
- `src/context/PopupContext.tsx`
- `src/components/MasteryPopupWrapper.tsx`

Current pages include:

| Route | Purpose |
| --- | --- |
| `/` | Home/feed/dashboard surface |
| `/goals` | Goal list and goal creation |
| `/goals/[goalId]` | Goal detail with sessions |
| `/search` | Global/user/post/activity search |
| `/settings` | User settings |
| `/u/[username]` | Public profile |
| `/u/edit` | Profile editing |
| `/a/[activity]` | Activity detail |
| `/a/[activity]/sessions` | Activity sessions |
| `/leaderboard/rookie` | Rookie leaderboard |
| `/leaderboard/goals` | Goal leaderboard |
| `/leaderboard/goals/[goalId]` | Goal-specific leaderboard |

### Fullscreen Shell

Routes under `app/(fullscreen)` use a minimal full-height layout without the sidebar.

| Route | Purpose |
| --- | --- |
| `/users/login` | Login |
| `/users/register` | Registration |
| `/goals/[goalId]/session/[sessionId]` | Live session timer |
| `/goals/[goalId]/session/[sessionId]/reflection` | Completed session reflection |

### Middleware

`middleware.ts` redirects unauthenticated requests to `/users/login`.

The matcher excludes:

- `/users/login`
- `/users/register`
- `/api/*`
- `/_next/*`
- `/favicon.ico`

Because route handlers are excluded, auth checks for API proxy calls happen inside each `app/api/**/route.ts` file.

## Providers and App State

`app/providers.tsx` wraps the app with:

- `ConvexProvider`, using `NEXT_PUBLIC_CONVEX_URL`
- `ThemeProvider`, using `next-themes`

The main and fullscreen layouts then wrap their content with `AuthProvider`.

### Auth Context

`src/context/AuthContext.tsx` exposes:

- `me`
- `loading`
- `refreshMe()`
- `logout()`

`refreshMe()` calls `/api/auth/me`, not Django directly. That route handler validates or refreshes cookies before returning the current user.

Use `useAuth()` inside client components when you need the logged-in user's ID or username.

## Authentication Flow

The frontend stores backend JWTs in HTTP-only cookies.

### Login

1. Login page posts credentials to `/api/auth/login`.
2. `app/api/auth/login/route.ts` forwards credentials to Django at `/api/v1/auth/login/`.
3. On success, the route handler sets `access` and `refresh` cookies.

### Current User

1. Client calls `/api/auth/me`.
2. Route handler sends the `access` token to Django `/api/v1/auth/me/`.
3. If Django returns `401`, the handler refreshes once via `refreshTokens()`.
4. If refresh succeeds, cookies are updated and the request is retried.
5. If refresh fails, cookies are cleared and the client is treated as logged out.

### Refresh Lock

`src/lib/auth/refreshLock.ts` ensures concurrent route handlers share a single refresh request instead of stampeding Django.

### Proxy Pattern

Most authenticated route handlers follow this shape:

1. Read `access` from `cookies()`.
2. Send request to Django with `Authorization: Bearer <access>`.
3. If the response is not `401`, return it.
4. On `401`, call `sharedRefresh(refreshTokens)`.
5. Retry once with the new access token.
6. Return JSON using the original backend status.

If you add new Django-backed endpoints, keep this behavior consistent.

## Backend Integration

The web app talks to Django through local Next route handlers instead of calling Django from most client components. This keeps cookies HTTP-only and keeps refresh logic on the server side.

Base URL:

```text
process.env.NEXT_PUBLIC_API_BASE_URL
```

Backend route convention:

```text
${NEXT_PUBLIC_API_BASE_URL}/api/v1/<resource>/
```

Frontend route convention:

```text
/api/<resource>
```

Examples:

| Frontend route | Django route |
| --- | --- |
| `/api/goals` | `/api/v1/goals/` |
| `/api/goals/[id]` | `/api/v1/goals/<id>/` |
| `/api/search` | `/api/v1/search/` |
| `/api/sessions` | `/api/v1/sessions/` |
| `/api/sessions/[sessionId]` | `/api/v1/sessions/<session_id>/` |
| `/api/sessions/calculate-rates` | `/api/v1/sessions/calculate-rates/` |

Some older page code still calls `NEXT_PUBLIC_API_BASE_URL` directly from client components. Prefer the Next route handler proxy for new work, especially when auth is required.

## Convex Session System

Convex is used for real-time session state. Django stores goals, users, activities, and lightweight session metadata.

Current principle:

- Convex is the source of truth for live and completed session runtime data.
- Django owns goal metadata, user identity, activity metadata, XP ledger updates, and mobile-friendly lightweight session lookups.

Primary files:

- `convex/schema.ts`
- `convex/sessions.ts`
- `convex/sessionJobs.ts`
- `convex/crons.ts`
- `src/lib/hooks/useConvexSessions.ts`
- `src/lib/hooks/useGoals.ts`
- `app/(fullscreen)/goals/[goalId]/session/[sessionId]/page.tsx`

### Convex Session Shape

The `sessions` table stores:

- Ownership: `userId`, `goalId`, `activityId`
- Denormalized activity metadata: `activityName`, `activityEmoji`, `activityType`
- Lifecycle: `live`, `paused`, `completed`
- Timing: `startedAt`, `endedAt`, `lastHeartbeatAt`, `pauseIntervals`
- Durations: total and focused seconds
- XP rates and totals: `rateSegments`, `xpTotal`, `xpBreakdown`
- Social metadata: `nudgeCount`
- Device context
- Completion metadata
- Django sync flags: `syncedToDjango`, `lastSyncedAt`

Indexes:

- `by_user`
- `by_user_status`
- `by_goal`
- `by_sync`
- `by_heartbeat`

### Session Lifecycle

```text
new session route
  -> fetch goal from Django
  -> calculate XP rates through /api/sessions/calculate-rates
  -> start Convex session
  -> fire-and-forget POST /api/sessions to register lightweight Django metadata
  -> heartbeat every 5 seconds while live
  -> pause/resume via Convex mutations
  -> complete or abandon via Convex mutation
  -> PUT /api/sessions/[sessionId] to sync final totals to Django
  -> mark Convex session synced
  -> redirect to reflection page
```

### Convex Mutations

`convex/sessions.ts` defines:

| Mutation | Purpose |
| --- | --- |
| `startSession` | Creates a live session after checking the user has no live or paused session |
| `heartbeat` | Recalculates duration and XP while live |
| `pauseSession` | Moves live to paused and opens a pause interval |
| `resumeSession` | Closes the open pause interval and returns to live |
| `completeSession` | Finalizes a live or paused session with `manual`, `auto`, or `timeout` reason |
| `abandonSession` | Finalizes as abandoned |
| `markSyncedToDjango` | Marks final payload as synced |

Queries:

| Query | Purpose |
| --- | --- |
| `getActiveSession` | Returns a live or paused session for a user |
| `getSession` | Returns one session by Convex ID |
| `getSessionsByGoal` | Returns all sessions for a goal in descending order |

### Cron Jobs

`convex/crons.ts` schedules:

- `cleanup stale sessions` every 2 minutes
- `retry django sync` every 5 minutes

`cleanupStaleSessions` completes live sessions with stale heartbeats after 5 minutes.

`retryDjangoSync` currently logs completed unsynced sessions. The HTTP action that retries Django sync is still a TODO.

## Goals and Sessions in React

`src/lib/services/goals.ts` normalizes goal and session payloads from the API. It accepts several backend response shapes, including raw arrays and paginated objects.

`src/lib/hooks/useGoals.ts` is the hybrid goal hook:

- Goal metadata comes from Django through `GoalsService.getGoal`.
- Session data comes from Convex through `useConvexSessionsByGoal`.
- Convex session documents are transformed into the existing Django-style `Session` interface.

This preserves compatibility with components originally built around Django session payloads.

For new goal detail UI, prefer `useGoal(goalId)` unless there is a specific reason to call Convex or Django directly.

## Search

Search helpers live in:

- `lib/api/search.ts`
- `src/lib/hooks/useSearch.ts`
- `src/lib/hooks/useSearchHistory.ts`
- `app/api/search/**`

Supported search surfaces:

- Global search across posts, users, and activities
- Posts-only search
- Users-only search
- Activities-only search
- Search history save, delete, and clear

`useSearch` debounces requests, aborts stale requests, normalizes image fields, and can save search history on first-page searches.

## UI and Styling

Global styling lives in `app/globals.css`.

Important CSS variable groups:

- Base app colors: `--background`, `--foreground`, `--border`, `--card`, `--muted`
- Dark surfaces: `--dark-1`, `--dark-2`, `--dark-3`
- Mastery colors: `--rookie-primary`, `--warrior-primary`, etc.
- Aspect colors: `--aspect-physique`, `--aspect-energy`, etc.

The app uses Tailwind v4 via:

```css
@import "tailwindcss";
@custom-variant dark (&:where(.dark, .dark *));
```

When adding UI:

- Prefer existing components in `src/components`.
- Keep aspect and mastery colors sourced from shared constants or CSS variables.
- Keep route pages thin when possible and move reusable behavior into `src/lib/hooks` or `src/lib/services`.
- For authenticated data, prefer calling local `/api/*` routes rather than Django directly.

## Adding a Django-Backed Feature

Use this flow for new REST-backed UI:

1. Add or confirm the Django endpoint.
2. Create a Next route handler in `app/api/<feature>/route.ts`.
3. Use the shared auth refresh pattern from existing route handlers.
4. Add a typed browser helper in `lib/api` or `src/lib/services`.
5. Add a hook in `src/lib/hooks` if the feature has loading, pagination, debounce, or normalization concerns.
6. Build UI using the local helper or hook.
7. Run `npm run lint` and, if feasible, `npm run build`.

Avoid calling Django directly from a client component unless the endpoint is public and intentionally unauthenticated.

## Adding Convex Session Behavior

When changing session runtime behavior:

1. Update `convex/schema.ts` first if storage changes are needed.
2. Update mutations and queries in `convex/sessions.ts`.
3. Regenerate Convex types with `npx convex dev` or the appropriate Convex command.
4. Update client hooks in `src/lib/hooks/useConvexSessions.ts`.
5. Update Django sync payloads in session timer code and matching `app/api/sessions/**` route handlers.
6. Update `CONVEX_DJANGO_INTEGRATION.md` and `TESTING_CONVEX_INTEGRATION.md` if data flow or validation changes.

Do not break the state machine:

```text
live <-> paused -> completed
live -> completed
```

Invalid transitions should throw clear errors.

## Testing and Verification

Current scripts:

```bash
npm run lint
npm run build
```

There is no dedicated test script in `package.json` yet.

For Convex session changes, also run a manual integration check:

1. Start Django locally.
2. Start Next with `npm run dev`.
3. Start Convex with `npx convex dev`.
4. Login.
5. Create or open a goal.
6. Start a session.
7. Confirm the session appears on the goal detail page without refresh.
8. Pause and resume.
9. Complete the session.
10. Confirm reflection loads and final totals sync to Django.
11. Open the same goal in another tab and verify real-time updates.

See `TESTING_CONVEX_INTEGRATION.md` for the deeper checklist.

## Deployment Checklist

Before shipping frontend changes:

- `NEXT_PUBLIC_API_BASE_URL` points to the production Django API.
- `NEXT_PUBLIC_CONVEX_URL` points to the correct Convex deployment.
- Convex schema and functions are deployed with `npx convex deploy`.
- Django has required endpoints and migrations for any changed payloads.
- `npm run lint` passes.
- `npm run build` passes.
- Session start, heartbeat, completion, and reflection flows are checked on staging.

## Known Gaps and Watch Items

- `convex/sessionJobs.ts` has a TODO for actual Django retry sync. It currently logs unsynced completed sessions.
- Some client pages still read `NEXT_PUBLIC_API_BASE_URL` directly. Prefer local Next route handlers for new authenticated calls.
- `AuthContext.tsx` has an unused `authResolved` state value.
- The codebase has duplicate sidebar component paths. Confirm which one is mounted before editing.
- Both npm and Yarn lockfiles are present. Avoid accidental updates to both.
- Some debug logging is present in search/discover/goals flows. Keep or remove intentionally depending on environment needs.
