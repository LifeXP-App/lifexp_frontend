## Project Summary

LifeXP is a gamified personal growth social app. Users create goals, start timed sessions against activities, complete reflections, and earn XP across five life aspects:

- `physique`
- `energy`
- `logic`
- `creativity`
- `social`

The frontend is a Next.js App Router application that talks to:

- a Django REST API through Next route handlers under `app/api/**`
- Convex for real-time session state under `convex/**`

The app uses HTTP-only cookies for Django JWTs. Client components should generally call local `/api/*` routes, not Django directly, unless the existing page already does so and the change is intentionally scoped.

## Middleware and Auth

`middleware.ts` redirects unauthenticated users to `/users/login` when no `access` cookie exists.

Excluded from middleware:

- `/users/login`
- `/users/register`
- `/api/*`
- `/_next/*`
- `/favicon.ico`

Because API routes are excluded, route handlers must enforce auth themselves.

### Auth Context

`src/context/AuthContext.tsx` provides:

```ts
me: { id: number; username: string } | null
loading: boolean
refreshMe(): Promise<void>
logout(): Promise<void>
```

`refreshMe()` calls `/api/auth/me`. Do not call Django `/api/v1/auth/me/` from client components for current-user state.

### Token Refresh

Server route handlers use:

- `src/lib/auth/refreshTokens.ts`
- `src/lib/auth/refreshLock.ts`
- occasionally `src/lib/auth/serverAuthFetch.ts`

`sharedRefresh(refreshTokens)` prevents concurrent requests from stampeding the Django refresh endpoint.

When adding an authenticated route handler, follow this pattern:

1. Read `access` from `cookies()`.
2. Return `401` if no token exists.
3. Forward to Django with `Authorization: Bearer <access>`.
4. If Django returns non-401, return that response body/status.
5. If Django returns 401, call `sharedRefresh(refreshTokens)`.
6. Retry once with the new access token.
7. Return `SESSION_EXPIRED` or `401` if refresh fails.

### Goals List Page

`app/(main)/goals/page.tsx`:

- fetches all goals from `/api/goals`
- splits into planned, ongoing, paused, completed, abandoned
- uses `GoalCard` for each status section
- opens `NewGoalModal` for creation
- opens `NewActivityModal` / "Pick Activity" when starting a new session
- uses optimistic creation and deletion
- uses `StatusChangeConfirmationModal` before status changes

Important current behavior:

- New goal creation inserts an optimistic planned item immediately.
- If the POST fails or returns non-OK, the optimistic item is removed.
- If POST succeeds but refresh fails, the optimistic/created item remains visible.
- `New Session` on ongoing goals opens the activity picker.
- Activity selection routes to `/goals/<goalId>/session/new?activity=<activityId>`.

### Goal Status Menu

`src/components/goals/GoalStatusMenu.tsx`:

- planned goals no longer show "Mark as Ongoing"
- ongoing goals can pause, complete, or abandon
- paused goals can resume, complete, or abandon
- completed and abandoned states have no menu actions
- normal dropdown text is black/white; abandon is red

### Goal Detail Page

`app/(main)/goals/[goalId]/page.tsx` uses `useGoal(goalId)`.

`useGoal` fetches:

- goal metadata from Django via `GoalsService.getGoal`
- session list in real time from Convex via `useConvexSessionsByGoal`

The hook transforms Convex session documents into the older Django-shaped `Session` interface for compatibility.

## Sessions and Convex

Sessions are hybrid:

- Django stores goal/activity/user metadata and session records for backend persistence.
- Convex stores real-time timer state, heartbeat, pause intervals, focused duration, and live XP totals.

Primary files:

- `convex/schema.ts`
- `convex/sessions.ts`
- `convex/sessionJobs.ts`
- `convex/crons.ts`
- `src/lib/hooks/useConvexSessions.ts`
- `app/(fullscreen)/goals/[goalId]/session/[sessionId]/page.tsx`
- `app/(fullscreen)/goals/[goalId]/session/[sessionId]/reflection/page.tsx`

Convex `sessions` fields include:

- ownership: `userId`, `username`, `goalId`, `goalTitle`, `activityId`
- denormalized activity data: `activityName`, `activityEmoji`, `activityType`
- lifecycle: `live`, `paused`, `completed`
- timing: `startedAt`, `endedAt`, `lastHeartbeatAt`, pause intervals
- totals: `totalDurationSeconds`, `focusedDurationSeconds`, `xpTotal`, `xpBreakdown`
- sync state: `syncedToDjango`, `lastSyncedAt`

Convex mutations:

- `startSession`
- `heartbeat`
- `pauseSession`
- `resumeSession`
- `completeSession`
- `abandonSession`
- `markSyncedToDjango`

Convex queries:

- `getActiveSession`
- `getSession`
- `getSessionsByGoal`
- `getLiveSessionsForActivity`

Cron jobs:

- cleanup stale sessions every 2 minutes
- retry Django sync every 5 minutes

Do not edit `convex/_generated/*` directly. Regenerate with Convex tooling.

## Data Normalization Rules

Backend payloads are not perfectly stable. Existing code normalizes multiple shapes.

Examples:

- goals may use `description` or `content`
- lists may arrive as direct arrays, `results`, `data`, `items`, or domain keys like `activities`
- profile images may be `profile_picture`, `profile_pic`, or `profile_picture_url`
- post images may be `post_image` or `post_image_url`
- activity search returns `{ activities, pagination }`

When adding data handling:

1. Inspect the actual caller and existing nearby normalization.
2. Accept known alternate shapes when low risk.
3. Do not let a successful mutation be treated as failed only because the response body is shaped differently.
4. For optimistic UI, rollback only on network failure or non-OK response unless the user asks otherwise.
